## 遇到的问题和解决
1. **问题**：在前端配置完 Axios 拦截器处理后，直接使用传统的 JSON Server 无法天然返回契约文档中规定的 `{ "code": 200, "msg": "xx", "data": {} }` 的标准统一格式，导致前端解析永远报出非此格式的异常。
   **解决**：放弃局限的方案，引入更高阶的加分利器 **MSW (Mock Service Worker)**。通过拦截浏览器层面的 Axios fetch 操作，在 `handlers.js` 手柄逻辑中自定义完美的 HTTP Body 包裹返回格式，完美融合了 API 约束。
2. **问题**：在进行 Apifox API 自动化功能测试生成“校验响应”结果时，由于之前 `api.yaml` 里面的 HTTP `201` 状态只写了描述缺少对象定义结构声明，被 Apifox 高级严格模式报出“该响应不应该有 Body”的错误红叉提示。
   **解决**：定位到严谨的结构性限制后，回到接口参数配置里的 “响应（Responses）”，为并补齐赋予对应的内容类型 （`Content-Type: application/json` 及对象形式的响应结构），确保了所有测试断言项 100% 绿色 `Passed` 完美运转。

## 心得体会
在本次作业的实战驱动下，我深入理解了作为前后端分离的关键所在——“API 契约 (Contract) 驱动开发”。
首先，规范的 OpenAPI 文件不仅为团队提供了最直观的手册文档，还可以借助像 Apifox 这样的工具自动化生成本地 Mock 假数据与多场景测试用例断言，不再是简单的连通性请求，更能够精确捕获不稳定的边界情况（尤其是参数类型的验证控制）。
通过从头配制 MSW 在前端挂载与调试的过程，我能够完全独立于后端队友随时随地地启动测试前端逻辑，大大降低了对依赖的耦合，为以后的中大型项目架构能力打下了异常重要的基石。
